Контекст и границы SoI

1. Тема

Платформа дистанционного обучения с прокторингом “Большой Брат”

2. Цель системы

Обеспечить процесс контролируемого дистанционного обучения:

  • предоставить обучающимся образовательную среду с доступом к учебным материалам и возможностью проходить аттестации
  • минимизировать риски нарушений во время проведения аттестаций
  • автоматизировать контроль за честностью проведения аттестаций
  • предоставить образовательной структуре среду для орагнизации учебной деятельности с возможностью отслеживания результатов

3. Внешние сущности

Для демонстрации потоков взаимодейтсвия между проектируемой системой и внешними сущностями воспользуемся контекстным уровнем диаграммы С4:

3.1 Пользователи и их взаимодействие с системой

  1. Студент
  • Вход в систему
  • Просмотр материалов по курсу
  • Выполнение заданий
  • Прохождение экзаменов
  1. Преподаватель
  • Вход в систему
  • Создание и редактирование курсов
  • Оценка заданий и экзаменов
  • Просмотр отчетов успеваемости
  • Просмотр отчетов проведения экзамена
  1. Проктор
  • Вход в систему
  • Контроль процесса проведения экзамена
  • Анализ видеопотока
  • Анализ сохраненных экзаменов
  1. Администратор учебного подразделения
  • Назначение ролей
  • Управление учетными записями
  • Настройка параметров экзаменов

3.2 Внешние интеграции и их обоснование

  1. Интеграция с системой единого входа (SSO)
  • Студенты и преподаватели используют единые корпоративные учётные данные. Нет необходимости запоминать ещё один логин и пароль.
  • Учётная запись в платформе создаётся автоматически при первом входе, синхронизация атрибутов (ФИО, email, группа) происходит из SSO
  1. Интеграция с LMS (Moodle)
  • Moodle содержит мощный, отлаженный движок тестирования: десятки типов вопросов, гибкие настройки оценивания и поведения теста. Разработка аналогичной системы “с нуля” потребует дополнительных времнных и трудовых ресурсов, которые рационально потратить на проработку модуля прокторинга.
  • Организации, использующие Moodle как основную LMS, могут не согласиться переносить курсы и тесты в новую систему.
  1. Интеграция с облачным объектным хранилищем
  • Использование сертифицированных российских облаков (Яндекс.Облако, VK Cloud) с аттестацией ФСТЭК позволяет выполнить требования 152-ФЗ о хранении персональных данных на территории РФ и использовании сертифицированных средств защиты.
  • Провайдеры (Яндекс.Облако, VK Cloud, AWS S3) обеспечивают репликацию данных в нескольких дата-центрах, гарантируя сохранность даже при выходе из строя оборудования.
  1. Интеграция с SFU-сервером
  • Организация низколатентной видеосвязи с десятками одновременных участников для бесперебойного контроля проведения экзамена

4. Границы системы

Для демонстрации того, что лежит внутри границ системы приведен концепт диаграммы С4 на уровне контейнеров (без углубления в технологии, поскольку их определению посвящаена работа 4)

4.1. В зоне отвественности

  1. Веб-приложение платформы - обеспечивает интерфейс доступа к курсам, материалам для студентов, управление курсами для преподавателей, контроль за проведением экзамена для прокторов, контроль учебной деятельности для администратора образовательной организации
  2. Настольное приложение для проведения проведения экзамена - обеспечивает честность проведения экзамена, отслеживая действия пользователя в системе.
  3. Сервис Прокторинга - модуль контроля проведения тестирования
  4. Основной Backend - модуль для организации учебного процесса

4.2. Вне зоны отвестнности

  1. Операции аутентификации и управления идентичностью пользователей выполняются внешней системой аутентификации (SSO). Система использует только результаты аутентификации, предоставленные SSO, для управления доступом к своим функциям.
  2. Организация учебных курсов, модулей и тестировние производится через интеграцию с LMS Moodle. Реализация собственной LMS не подразумевается по уже представленным выше причинам
  3. Физическое размещение видеоданных, аппаратная отказоустойчивость и резервное копирование осуществляется через интеграцию с облачным объектным хранилищем. Реализация собственной подобной инфраструктуры не предусматривается
  4. Организацией кодирования/декодирования видеопотоков занимается SFU-сервер. Проектирование собственной системы передачи и обработки видеопотока не осуществляется

Сценарии использования системы прокторинга

Сценарий 1. Вход студента в систему

Цель

Вход пользователя в систему с назначением корректной роли.

Условия начала

  1. Пользователь зарегистрирован.
  2. SSO доступен.
  3. Аккаунт не заблокирован.

Основной поток (Happy path)

  1. Пользователь открывает платформу и выбирает “Войти”.
  2. Система запрашивает сведения через SSO.
  3. SSO возвращает токен.
  4. Система назначает роль.
  5. Система открывает личный кабинет.

Результат

Пользователь авторизован и имеет доступ к ресурсам в соответсвии с его ролью

Исключения (нештатные ситуации)

  1. Неверные данные авторизации
  • 1.1 SSO не вернула токен.
  • 1.2 Платформа выдаёт ошибку.
  1. Токен пользователя истёк
  • 2.1 Платформа повторяет запрос на получение токена
  1. Пользователь заблокирован
  • 3.1 Платформа отклоняет авторизацию и выдаёт сообщение о блокировке.

Режимы деградации (как продолжаем работать)

  1. SSO недоступен
  • 1.1 Система должна запретить выполнение процессов повторной аутентификации через SSO
  • 1.2 Система должна предоставлять доступ только на основании ранее созданной и действующей сессии пользователя
  • 1.3 Система не должна создавать новые учётные записи или сессии
  • 1.4 Система должна ограничивать доступ до функциональности, не требующей повторной аутентификации

Сценарий 2. Проведение экзамена с прокторингом

Цель

Проведение экзамена с обязательным видеомониторингом и фиксацией сессии.

Условия начала

  1. Студент авторизован.
  2. Экзамен доступен для выполнения.
  3. Устройство студента имеет доступ к камере и микрофону.

Основной поток (Happy path)

  1. Студент открывает страницу экзамена.
  2. Студент соглашения с условиями проведения экзамена.
  3. Запрашивается доступ к камере/микрофону.
  4. Студент разрешает доступ к камере и микрофону
  5. Студент нажимает кнопку “Начать экзамен”.
  6. Создаётся прокторинг-сессия.
  7. Видеопоток подключается к стриминг-сервису.
  8. Включается античит-система.
  9. Студент получает экзаменационные задания.
  10. Запускается таймер.
  11. Студент выполняет задания.
  12. Записи камеры, микрофона, экрана фиксируются, ответы студента сохраняются
  13. Результаты экзамена сохраняются вместе с записями.

Результат

Экзамен проведен, данные сохранены, прокторинг-сессия зарегистрирована.

Исключения (нештатные ситуации)

  1. Камера или микрофон недоступны
  • 1.1 Система блокирует начало экзамена.
  • 1.2 Кнопка начать экзамен недоступна
  1. Потеря интернет-соединения
  • 2.1 Система детектирует потерю связи и уведомляет студента: “Соединение потеряно. Экзамен продолжается в offline-режиме.”
  • 2.2 Все действия студента сохраняются локально
  • 2.3 При восстановлении соединения данные синхронизируются с сервером.

Режимы деградации

  1. Сервис стриминга недоступен
  • 1.1 Разрешить запись локально на устройство студента с последующей загрузкой.
  • 1.2 Создать хеш видео-записи и подписать его для обеспечения неподдельности видеозаписи
  1. SSO недоступен во время экзамена
  • 2.1 Продолжать выполнение экзамена на основании ранее установленной и действующей сессии пользователя
  • 2.2 Сохраняем возможность ввода, сохранения и отправки ответов в рамках текущего экзамена
  • 2.3 Ограничиваем доступ пользователя только функциональностью текущего экзамена без доступа к другим частям системы
  • 2.4 Сохраняем все результаты и промежуточные ответы до завершения экзаменационной сессии

Сценарий 3. Обнаружение нарушения во время экзамена

Цель

  1. Зафиксировать и обработать потенциальное нарушение.

Условия начала

  1. Экзамен в процессе.
  2. Проктор доступен (онлайн-прокторинг)
  3. Подозрительное событие со стороны экзаменуемого.

Основной поток (Happy path)

  1. Система обнаруживает подозрительное событие (второе лицо в кадре, переключение вкладки, уход из кадра).
  2. Система фиксирует потенциальное нарушение с описанием и временем.
  3. Проктор получает уведомление.
  4. Проктор просматривает фрагмент видео.
  5. Проктор принимает решение и при необходимости шлет предупреждение экзаменуемому
  6. Решение фиксируется в системе.

Результат

Событие зарегистрировано, решение зафиксировано.

Исключения (нештатные ситуации)

  1. Ложное срабатывание Античита
  • 1.1 Проктор меняет статус потенциального нарушения на “Ложное”.
  1. Массовые события (перегрузка)
  • 2.1 Система расставляет приоритеты потенциальным нарушениям по уровню риска.

Режимы деградации

  1. Проктор недоступен
  • 1.1 Решение по нарушениям принимается после экзамена при проверке записи (офлайн-прокторинг).

Сценарий 4. Управление политиками прокторинга администратором

Цель

Настроить правила прокторинга и политики безопасности.

Условия начала

  1. Администратор учебного подразделения авторизован.

Основной поток (Happy path)

  1. Администратор учебного подразделения открывает “Настройки прокторинга”.
  2. Настраивает параметры: допустимое количество переключений вкладок, чувствительность античита.
  3. Сохраняет изменения.
  4. Новые изменения логируются (кто внес, время, старое и новое значение параметра)
  5. Система применяет новые параметры.
  6. Параметры применяются к новым экзаменам.

Результат

Настройки прокторинга обновлены и активны.

Исключения (нештатные ситуации)

  1. Конфликт параметров
  • 1.1 Система уведомляет о конфликте и требует исправления.

Режимы деградации

  1. Античит недоступен
  • 1.1 Параметры античита отправляются в очередь
  • 1.1 Когда сервис с античитом станет доступен, параметры из очереди будут отправлены античиту

Сценарий 5. Апелляция студента по результатам экзамена

Цель

  1. Обеспечить прозрачный процесс пересмотра результатов.

Условия начала

  1. Экзамен завершён.
  2. Студент имеет основания для подачи апелляции.

Основной поток (Happy path)

  1. Студент нажимает “Подать апелляцию”.
  2. Указывает причину и прикрепляет комментарии.
  3. Система фиксирует заявку.
  4. Преподаватель (или администратор) получает уведомление.

Результат

Апелляция студента зарегистрирована и передана на рассмотрение.

Исключения (нештатные ситуации)

  1. Видео отсутствует
  • 1.1 Решение по апелляции принимается на основе доступных логов и данных.

Режимы деградации

  1. БД платформы недоступна
  • 1.1 Апелляция ставится в очередь и отправляется автоматически после восстановления работы БД.

Требования к системе с т.з. стейкхолдеров (SR-SH)

SR-SH-1 (Студент - доступ к обучению и экзаменам)

Система должна предоставлять студенту доступ к учебным материалам, заданиям, тестам и экзаменам в рамках назначенных ему дисциплин внутри единой системы.

SR-SH-2 (Студент - понятность и предсказуемость экзамена)

Система должна обеспечивать студенту понятный сценарий прохождения экзамена, включая правила, текущее состояние попытки и результат завершения, чтобы снизить количество ошибок и спорных ситуаций.

SR-SH-3 (Студент - защита результатов при сбоях)

Система должна гарантировать студенту, что результаты его экзаменов и выполненных заданий сохраняются даже при технических сбоях системы или потере соединения, чтобы его работа не была утрачена и не требовала повторного выполнения.

SR-SH-4 (Студент - прозрачность прокторинга и право на оспаривание)

Система должна предоставлять студенту информацию о факте прокторинга, фиксируемых событиях и основаниях для возможных замечаний, а также возможность инициировать разбор спорных случаев, чтобы процедура контроля была прозрачной и справедливой.

SR-SH-5 (Преподаватель - управление контентом)

Система должна позволять преподавателю публиковать и актуализировать учебные материалы, задания, тесты и экзамены, чтобы поддерживать курс в актуальном состоянии на протяжении учебного процесса.

SR-SH-6 (Преподаватель - контроль результатов обучения)

Система должна предоставлять преподавателю результаты студентов по заданиям, тестам и экзаменам в форме, достаточной для оценки знаний и принятия решений по аттестации.

SR-SH-7 (Преподаватель - работа с инцидентами прокторинга)

Система должна предоставлять преподавателю сведения о зафиксированных инцидентах для комплексной и справедливой оценки.

SR-SH-8 (Проктор - наблюдаемость экзаменационной сессии)

Система должна обеспечивать проктору возможность своевременно выявлять потенциальные нарушения во время экзаменационной сессии, чтобы обеспечивать соблюдение правил проведения экзамена.

SR-SH-9 (Министерство Образования - соответствие регламенту)

Система должна соответствовать стандартам министерства для обспечения целей образовательной деятельности

SR-SH-10 (Администратор учебной организации - управление доступом и ролями)

Система должна позволять администратору учебной организации управлять учётными записями пользователей, их ролями и доступом к функциям системы, чтобы обеспечивать корректную и безопасную эксплуатацию платформы.

SR-SH-11 (Поставщик платформы - управляемость работы платформы)

Система должна давать поставщику платформы возможность контролировать состояние и стабильность работы сервиса, чтобы обеспечивать его надёжную эксплуатацию и выполнение обязательств по доступности.

SR-SH-12 (Образовательная организация - интеграция в существующую инфраструктуру)

Образовательная организация должна иметь возможность использовать систему совместно с существующими образовательными и административными сервисами (например, учётные системы SSO и LMS), чтобы избежать необходимости ведения дублирующих данных и обеспечить единый контур управления обучением.

SR-SH-13 (Образовательная организация - проведение обучения и аттестации)

Система должна поддерживать проведение дистанционного обучения, тестирования и экзаменов, чтобы учебный процесс мог осуществляться централизованно и контролируемо.

SR-SH-14 (Образовательная организация - аналитика и отчётность)

Система должна предоставлять образовательной организации сводную информацию об успеваемости, активности обучающихся и итогах аттестации, чтобы руководство могло контролировать качество обучения и принимать управленческие решения.

SR-SH-15 (Юрист - правомерность и проверяемость)

Юрист должен иметь доступ к информации, достаточной для подтверждения законности обработки персональных данных, проведения экзаменов и хранения результатов, чтобы обеспечивать правовую защиту образовательной организации.

Системные требования (SR-SYS)

SR-SYS-1 (Сценарий 1 - Переход в SSO)

При инициировании пользователем входа в систему, система должна перенаправлять пользователя в систему аутентификации (SSO).

Верификаиция: функциональный тест (инициирование входа => пользователь перенаправлен в SSO)

SR-SYS-2 (Сценарий 1 - Успешный вход в систему)

При успешной аутентификации пользователя в SSO система должна принимать и обрабатывать токен доступа, возвращённый SSO.

Верификация: функциональный тест (SSO возвращает валидный токен => система обрабатывает токен и продолжает процесс авторизации).

SR-SYS-3 (Сценарий 1 - Безуспешный вход в систему)

При безуспешной аутентификации пользователя в SSO система не должна создавать пользовательскую сессию, уведомив пользователя об ошибке аутентификации.

Верификация: функциональный тест (SSO возвращает ошибку аутентификации => пользователь не авторизован => отображено сообщение об ошибке)

SR-SYS-4 (Сценарий 1 - Назначение ролей)

При успешной аутентификации пользователя система должна определять и назначать ему роль на основании данных, полученных от SSO.

Верификация: функциональный тест (получен токен с атрибутами пользователя => система извлекает роль => пользователю доступны функции, соответствующие роли)

SR-SYS-5 (Сценарий 1 - Истечение токена: без повторной аутентификации)

При обнаружении истечения срока действия токена доступа пользователя система должна инициировать процесс получения нового токена через SSO и, при успешном получении нового токена, продолжать текущую пользовательскую сессию без повторной аутентификации.

Верификация: функциональный тест (истёкший токен => система выполняет запрос к SSO => получает новый токен => пользователь сохраняет авторизованную сессию без повторного входа)

SR-SYS-6 (Сценарий 1 - Истечение токена: с необходимостью повторной аутенфикации)

При невозможности автоматического получения нового токена через SSO система должна прекращать выполнение защищённых операций и инициировать повторную аутентификацию пользователя.

Верификация: функциональный тест (новый токен не получен => блокировка защищённых операций => редирект на SSO)

SR-SYS-7 (Сценарий 1 - Заблокированный пользователь)

Если пользователь имеет статус “заблокирован”, система должна отклонять авторизацию и запрещать выполнение любых операций от имени пользователя, даже при успешной аутентификации в SSO.

Верификация: функциональный тест (пользователь с валидными SSO-данными, но статусом “blocked” => сессия не создается, доступа к системе нет => сообщение о блокировке)

SR-SYS-8 (Сценарий 1 - Недоступность SSO)

При недоступности системы аутентификации (SSO) система должна запрещать выполнение операций, требующих полной аутентификации пользователя, и обеспечивать доступ только для пользователей с ранее установленной и действующей локальной сессией

Верификация: функциональный тест: SSO недоступен:

  • пользователь без локальной сессии => доступ запрещён
  • пользователь с валидной локальной сессией => ограниченный доступ предоставлен

SR-SYS-9 (Сценарий 2 - Допуск к старту экзамена)

Система должна разрешать запуск экзамена только авторизованному студенту в пределах открытого окна экзамена.

Верификация: функциональный тест (авторизованный студент + наступило время экзамена => запуск разрешён).

SR-SYS-10 (Сценарий 2 - Проверка попыток и доступа к экзамену)

При попытке запуска экзамена система должна проверять, что пользователь имеет право на прохождение экзамена и что количество доступных попыток не исчерпано, после чего либо разрешать запуск экзамена, либо отклонять его запуск.

Верификация: функциональный тест (нет доступа или попытки исчерпаны => запуск отклонён, есть доступ и попытки не исчерпаны => запуск разрешен).

SR-SYS-11 (Сценарий 2 - Запрос доступа к камере и микрофону)

Перед началом экзамена система должна инициировать запрос разрешений на доступ к камере и микрофону пользователя, необходимых для проведения прокторинга.

Верификация: функциональный тест (пользователь нажимает “Начать экзамен” => появляется запрос доступа к камере и микрофону).

SR-SYS-12 (Сценарий 2 - Проверка и блокировка старта без устройств)

Система должна разрешать начало экзамена только при наличии доступа к камере и микрофону, и должна блокировать запуск экзамена при отсутствии хотя бы одного из требуемых разрешений или недоступности устройств.

Верификация: функциональный тест (доступ к камере/микрофону отсутсвтует => старт заблокирован).

SR-SYS-13 (Сценарий 2 - Создание прокторинг-сессии)

После успешного прохождения всех проверок допуска к экзамену (включая аутентификацию, проверку попыток и доступность необходимых устройств) система должна создавать уникальную прокторинг-сессию, связанную с конкретным пользователем и экзаменом.

Верификация: функциональный тест (пользователь прошёл все проверки допуска => выполнено “Начать экзамен” => создана прокторинг-сессия).

SR-SYS-14 (Сценарий 2 - Подключение видеопотока)

После создания прокторинг-сессии и получения необходимых разрешений пользователя система должна устанавливать соединение с сервисом передачи медиапотоков и передавать видеопоток, связанный с экзаменационной сессией пользователя.

Верификация: интеграционный тест (создана прокторинг-сессия => видеопоток успешно передаётся в media-сервис).

SR-SYS-15 (Сценарий 2 - Включение античита)

После создания прокторинг-сессии система должна инициализировать античит, передавая идентификатор экзаменационной сессии и параметры наблюдения, для последующего анализа пользовательской активности в ходе экзамена.

Верификация: интеграционный тест (создана прокторинг-сессия => античит получает session_id и параметры конфигурации => античит инициализирован).

SR-SYS-16 (Сценарий 2 - Выдача вопросов и запуск таймера)

После успешного завершения всех этапов инициализации экзаменационной сессии система должна загрузить и отобразить студенту набор экзаменационных заданий и одновременно инициировать запуск серверного таймера экзамена.

Верификация: функциональный тест (экзамен начат => отображаются вопросы, запущен таймер).

SR-SYS-17 (Сценарий 2 - Фиксация записи камеры, микрофона и экрана)

Во время активной экзаменационной сессии система должна обеспечивать захват и сохранение медиапотоков (видеопоток с камеры, аудиопоток с микрофона и поток экрана пользователя) с привязкой к идентификатору экзаменационной сессии.

Верификация: функциональный тест (экзамен заупщен => создаются записи всех предусмотренных источников).

SR-SYS-18 (Сценарий 2 - Сохранение результатов вместе с записями)

После завершения экзаменационной сессии система должна сохранять результаты экзамена, медиозаписи (камера, микрофон, экран) и журнал событий прокторинга и античита как единый связанный набор данных, ассоциированный с идентификатором экзаменационной попытки.

Верификация: функциональный тест (экзамен завершен => сохранены ответы, видео и аудио, события прокторинга и связаны с уникальным индентификатором экзаменнационной сессии).

SR-SYS-19 (Сценарий 2 - Потеря интернет-соединения)

При потере сетевого соединения во время активной экзаменационной сессии система должна переводить экзамен в оффлайн-режим, обеспечивать локальное сохранение вводимых ответов и состояния таймера, а также сохранять возможность продолжения выполнения экзамена без прерывания пользовательской сессии.

Верификация: функциональный тест (разрыв соединения => система не завершает экзамен, сохраняет ответы локально,отображает статус offline).

SR-SYS-20 (Сценарий 2 - Недоступность стриминг-сервиса)

При недоступности SFU-сервиса система должна автоматически переключаться в режим локальной записи медиапотоков (камера, микрофон, экран) с их временным хранением на устройстве пользователя и последующей автоматической загрузкой в систему после восстановления соединения с SFU.

Верификация: интеграционный тест (SFU недоступен => запись продолжается локально и буферизует медиафайлы до момента восстановления).

SR-SYS-21 (Сценарий 2 - Недоступность SSO)

Если недоступность SSO возникает во время активной экзаменационной сессии, система должна обеспечивать непрерывность выполнения экзамена, включая ввод и сохранение ответов, без прерывания сессии пользователя, при условии действующей локальной экзаменационной сессии.

Верификация: функциональный тест:

  • пользователь начал экзамен
  • SSO становится недоступен
  • Система не завершает экзамен, продолжает отображать задания, сохраняет ответы, пользователь может завершить экзамен

SR-SYS-22 (Сценарий 3 - Регистрация подозрительного события)

При обнаружении подозрительного события любым компонентом системы (античит, прокторинг или анализ медиопотоков) система должна создавать запись инцидента нарушения, содержащую тип события, время фиксации, идентификатор экзаменационной сессии и ссылку на связанные артефакты (видео, экран, логи).

Верификация: функциональный тест (произвести моделирование подозрительного события => создается запись о событии с указанием типа, времени фиксации, идентификатора экзаменационной сессии и ссылкой на связанные артефакты).

SR-SYS-23 (Сценарий 3 - Уведомление проктора)

После регистрации инцидента система должна отправлять проктору уведомление в режиме близком к реальному времени, содержащее идентификатор экзаменационной сессии, тип события и ссылку на связанные артефакты (видео/экран/логи).

Верификация: функциональный тест (создаётся запись об инциденте => проктор получает уведомление).

SR-SYS-24 (Сценарий 3 - Доступ проктора к фрагменту записи)

Для зарегистрированного инцидента нарушения система должна предоставлять проктору доступ к связанному медиофрагменту, включающему видеопоток с камеры и синхронизированные аудио- и экранные данные, охватывающему временной интервал до и после момента фиксации события.

Верификация: функциональный тест (получено уведомление об инциленте => проктор открывает инцидент => система отображает видеофрагмент и временную шкалу события)

SR-SYS-25 (Сценарий 3 - Фиксация решения проктора)

Система должна позволять проктору фиксировать решение по зарегистрированному инциденту с указанием типа решения, комментария и подтверждающих артефактов. Решение должно быть привязано к инциденту и сохраняться в системе в неизменяемом виде с записью автора и времени принятия.

Верификация: функциональный тест (существует запись инцидента => проктор принимает решение => система сохраняет тип решения, время решения, идентификатор проктора, ссылку на инцидент).

SR-SYS-26 (Сценарий 3 - Подтверждённое нарушение)

Если проктор принимает решение о подтверждении нарушения, система должна отправить студенту уведомление, содержащее тип нарушения, время фиксации, комментарий проктора.

Верификация: функциональный тест (проктор подтверждает нарушение => студент получает уведомление).

SR-SYS-27 (Сценарий 3 - Массовые события)

При поступлении большого количества событий от систем мониторинга система должна агрегировать события, оценивать их уровень риска и формировать упорядоченную очередь инцидентов для обработки проктором на основе приоритета.

Верификация: нагрузочный/функциональный тест (создается N событий => система агрегирует связанные события в инциденты, присваивает уровень риска, формирует очередь приоритетов).

SR-SYS-28 (Сценарий 3 - Режимы деградации при недоступности проктора)

Если проктор недоступен в момент или после завершения экзаменационной сессии, система должна переводить результаты экзамена в состояние “ожидает проверки” и обеспечивать возможность принятия решения по результатам экзамена на основе сохранённых артефактов (ответы студента, медиозаписи и журналы событий).

Верификация: функциональный тест (проктор отсутствует и экзамен завершен => система сохраняет все артефакты и устанавливает статус “требует проверки”).

SR-SYS-29 (Сценарий 4 - Доступ администратора к настройкам прокторинга)

Система должна предоставлять доступ к разделу “Настройки прокторинга” только пользователям с ролью администратора учебного продразделения, проверяя права доступа на уровне пользовательской сессии и серверной авторизации. Все попытки доступа должны проходить проверку прав перед выполнением операции.

Верификация: функциональный тест (пользователь с ролью “Администратор уч. подразделения” имеет доступ к настройкам. Пользователь с иной ролью такого доступа не имеет).

SR-SYS-30 (Сценарий 4 - Настройка допустимых переключений вкладок)

Система должна позволять администратору учебного подразделения задавать максимальное допустимое количество переключений вкладок в рамках одной экзаменационной сессии, которое используется системой прокторинга для фиксации потенциальных нарушений и формирования событий риска.

Верификация: функциональный тест (администратор устанавливает значение => система сохраняет параметр => параметр применяется к новым экзаменам).

SR-SYS-31 (Сценарий 4 - Настройка срока хранения записей)

Система должна позволять администратору учебного подразделения задавать срок хранения экзаменационных данных (включая видеозаписи, аудиозаписи, экранные записи, ответы студентов и журналы событий).

Верификация: функциональный тест (администратор задаёт срок хранения => система сохраняет настройку).

SR-SYS-32 (Сценарий 4 - Политика хранения данных)

Система должна автоматически обеспечивать удаление или обезличивание экзаменационных данных по истечении установленного срока хранения.

Верификация: функциональный тест (истек срок хранения => данные становятся недоступны, выполняется удаление или обезличивание).

SR-SYS-33 (Сценарий 4 - Настройка параметров античита)

Система должна позволять авторизованному администратору учебного подразделения изменять ограниченный набор параметров античит-системы, используемых для обнаружения подозрительных действий в рамках экзаменационных сессий. Изменения должны фиксироваться, проходить валидацию и гарантированно доставляться в античит-сервис с подтверждением применения.

Верификация: функциональный тест (администратор изменяет параметр => система валидирует значение, сохраняет конфигурацию и отправляет в античит-сервис => античит получает параметр и использует в новых сессиях).

SR-SYS-34 (Сценарий 4 - Сохранение и применение новых политик)

Система должна обеспечивать версионирование политик прокторинга и применять актуальную версию политики только к экзаменационным сессиям, созданным после момента её публикации, сохраняя неизменность ранее назначенных политик для уже созданных экзаменов.

Верификация: функциональный тест:

  1. создаётся политика v1 => создаётся экзамен A => экзамен A использует v1
  2. изменяется политика => создаётся v2 => создаётся экзамен B => экзамен B использует v2
  3. Проверка:
  • экзамен A всегда использует v1
  • экзамен B всегда использует v2
  • изменение v2 не влияет на экзамен A

SR-SYS-35 (Сценарий 5 - Доступность подачи апелляции)

Система должна предоставлять студенту возможность подать апелляцию только по экзаменационной сессии, находящейся в статусе “завершена и оценена”, и только в пределах установленного временного окна после публикации результата.

Верификация: функциональный тест:

  1. экзамен в статусе “в процессе” или “ожидает проверки” => пелляция недоступна
  2. экзамен оценён => апелляция доступна
  3. истекло время подачи апелляции => апелляция недоступна

SR-SYS-36 (Сценарий 5 - Обязательность указания причины)

При подаче апелляции система должна создавать уникальную запись апелляции, связанную с результатом экзаменационной сессии, фиксировать обязательную причину обращения, и устанавливать начальный статус “зарегистрирована”.

Верификация: функциональный тест:

  1. попытка отправить апелляцию без причины => отклоняется системой
  2. попытка отправить с причиной => апелляция создаётся, причина фиксируется в неизменном виде, записи апелляции назначается уникальный индентификатор и статус “зарегистрирована”

NFR / качество с метриками (исправленная версия)

Общая методика оценки качества

В данной работе показатели качества оцениваются по трём группам метрик:

  1. Метрики производительности - например, p95 времени отклика для ключевых операций.
  2. Метрики доступности - SLA для критичных пользовательских функций платформы.
  3. Метрики точности и надёжности - доля ложных срабатываний, сохранность данных, целостность.

Под SLA понимается доля времени, в течение которого критичная функция системы была доступна пользователю в штатном или допустимом деградированном режиме.
Для платформы прокторинга SLA оценивается по функциям входа, запуска и прохождения экзамена, сохранения результатов, уведомления проктора, управления политиками и подачи апелляции.
Внешние системы (SSO, Moodle, SFU, облачное хранилище) рассматриваются как внешние зависимости: их отказ не считается нарушением SLA платформы, если система корректно переходит в предусмотренный режим деградации.

p95 используется для оценки времени отклика критичных операций и показывает верхнюю границу времени выполнения для 95% запросов.

NFR для системы онлайн-прокторинга и экзаменов

NFR-1 (Производительность входа через SSO - сценарий 1)

  • Время обработки токена и создания сессии (без учёта задержки SSO) не должно превышать 1,5 секунды в 95-м перцентиле при нагрузке до 300 одновременных попыток входа.
    Общее время входа для пользователя (с учётом задержки SSO) не должно превышать 6 секунд в 95-м перцентиле при задержке SSO <= 700 мс.
  • Доля неуспешных входов по внутренней ошибке платформы должна составлять не более 1% от общего числа попыток входа за календарный месяц.

Верификация:
Нагрузочное тестирование с не менее чем 5000 попытками входа при профиле нагрузки до 300 одновременных входов в течение не менее 10 минут. Дополнительно - эмуляция ответа SSO в диапазоне 200 - 700 мс. Результаты подтверждаются анализом логов аутентификации с расчётом p95 и p99 задержек, при этом доля ошибок с кодом 5xx должна быть < 1%.

NFR-2 (Доступность авторизации и защищённых операций)

  • Функции входа, обработки токена SSO и назначения роли должны быть доступны с коэффициентом доступности не ниже 99,5% в месяц, исключая запланированные окна обслуживания длительностью не более 4 часов в месяц.
  • При недоступности SSO система должна сохранять доступ для пользователей с валидной локальной сессией в >= 99,9% проверенных случаев, но не должна создавать новые сессии.

Верификация:
SLA-мониторинг на основе health-check с интервалом 30 секунд. fault-injection тесты с отключением SSO на 5 - 10 минут не менее чем в 5 повторениях. Проверяется сохранение активных сессий, которое должно быть >= 99,9%.

NFR-3 (Отклик запуска экзамена - сценарий 2)

  • От нажатия кнопки «Начать экзамен» до отображения экзаменационных заданий и запуска серверного таймера должно проходить не более 7 секунд в 95-м перцентиле при одновременном старте до 300 экзаменационных сессий и при условии штатной работы Moodle (p95 времени ответа <= 1,5 с).
    Включает: проверку доступа к устройствам, создание прокторинг-сессии, инициализацию античита, загрузку заданий.
  • Время создания прокторинг-сессии не должно превышать 3 секунд в 95-м перцентиле.

Верификация:
Нагрузочное тестирование с >= 300 одновременными запусками экзамена, при этом нагрузка удерживается не менее 10 минут в каждой из >= 3 тестовых серий. Выполняется анализ логов с расчётом p95 и p99 времени выполнения этапов. Успешность запуска экзамена должна быть >= 99%.

NFR-4 (Производительность пользовательских интерфейсов)

  • Загрузка основных страниц (список экзаменов, инциденты, отчёты) должна выполняться не более 5 секунд в 95-м перцентиле при типовой нагрузке (до 200 RPS API).

Верификация:
Нагрузочное тестирование при 200 RPS в течение 10 минут. Измерение времени полного ответа API и времени загрузки UI. Результаты подтверждаются p95 <= 5 секунд и уровнем ошибок <= 2%.

NFR-5 (Сохранность ответов при потере соединения)

  • При потере интернет-соединения изменения ответов должны сохраняться локально не позднее чем через 2 секунды после события изменения.
  • После восстановления соединения синхронизация несохранённых ответов и метаданных должна завершаться не позднее 120 секунд для разрыва связи до 10 минут при объёме изменений <= 1 МБ.
  • Потеря ответов допускается не более чем в 1% сессий, потеря финального результата - не более 0,1%.

Верификация:
fault-injection тестирование с разрывами соединения длительностью 1, 5, 10 и 15 минут не менее чем в 10 повторениях. Проверяется корректность локального сохранения данных и синхронизации после восстановления соединения. Контролируется полнота синхронизации >= 99% для разрывов до 10 минут, а расхождение таймера <= 2 секунды.

NFR-6 (Сохранность и целостность артефактов экзамена)

  • Ответы, записи и журналы событий должны сохраняться как единый связанный набор данных с успешной фиксацией не менее чем в 99,5% сессий.
  • Для локальных записей должен вычисляться хеш, а проверка целостности при загрузке должна быть успешной в >= 99,9% случаев.
  • Допустимый уровень утраты артефактов после подтверждённого сохранения - не более 0,1% сессий за месяц.

Верификация:
Тесты целостности с >= 20 случаями повреждения файлов. Выполняется аудит >= 500 сессий и выборка >= 1000 объектов в хранилище. Контролируется уровень потерь <= 0,1% подтверждённых сессий.

NFR-7 (Обработка и доставка событий прокторинга - сценарий 3)

  • От момента фиксации события до доставки уведомления проктору должно проходить не более 5 секунд в 95-м перцентиле при нагрузке до 1000 событий/мин, при всплесках до 3000 - не более 12 секунд.
  • Формирование очереди событий и инцидентов должно происходить не более чем за 45 секунд в 95-м перцентиле.
  • Доля потерянных событий - не более 0,2%.

Верификация:
Стресс-тестирование: 1000 событий/мин в течение 5 минут, затем 3000 событий/мин в течение 5 минут. Выполняется измерение задержки event -> UI. Контролируется доля обработанных событий >= 99,8%, а потери <= 0,2%.

NFR-8 (Надёжность интеграций)

  • При недоступности внешних систем должны выполняться не менее 3 повторных попыток с увеличивающейся задержкой.
  • При недоступности видеосервиса система должна переключаться на локальную запись за <= 3 секунд.
  • Доля сессий, прерванных из-за ошибок интеграций, - не более 0,3%.

Верификация:
fault-injection тестирование с отключением сервисов в >= 5 сценариях. Проверяется работа retry-механизма и логирование. Контролируется успешное завершение >= 99,7% тестовых сессий.

NFR-9 (Управляемость и применение политик прокторинга - сценарий 4)

  • Сохранение политики администратором должно занимать не более 3 секунд в 95-м перцентиле.
  • Доставка и применение параметров античитом - не более 60 секунд в 95-м перцентиле.
  • При недоступности античита изменения должны попадать в очередь и применяться после восстановления сервиса.

Верификация:
Интеграционные тесты >= 50 изменений политики, включая серии быстрых изменений и отказ античита на 5 минут. Проверяется, что >= 99% изменений применяются в пределах SLA и порядок применения сохраняется.

NFR-10 (Аудируемость и неизменяемость административных действий)

  • Для 100% изменений конфигурации система должна сохранять аудит-запись (пользователь, время, изменённые параметры).
  • Все административные операции должны проходить серверную проверку роли (RBAC).
  • Успешные случаи несанкционированного доступа - 0.
  • Аудит-логи должны быть неизменяемыми.

Верификация:
Проводится аудит >= 500 операций. Выполняется проверка наличия обязательных полей. Проводится >= 50 негативных тестов доступа. Все попытки изменения логов должны завершаться отказом.

NFR-11 (Доступность и устойчивость подачи апелляции - сценарий 5)

  • Подтверждение регистрации апелляции должно выполняться не позднее 3 секунд в 95-м перцентиле.
  • Доступность функции - не менее 99% в месяц.
  • При недоступности БД апелляции должны помещаться в очередь в >= 99,9% случаев и обрабатываться после восстановления.

Верификация:
Нагрузочные тесты >= 500 запросов и стресс до 1000 запросов. Выполняется отключение БД на 5 - 10 минут. Проверяется, что >= 99,9% запросов попадают в очередь и обрабатываются не позднее 15 минут.

NFR-12 (Точность античита)

  • Доля ложных срабатываний и пропущенных нарушений должна обеспечивать f1-score >= 85%.

Верификация:
Проводится выборка >= 1000 - 2000 сессий. Выполняется ручная разметка данных. Рассчитываются precision, recall и f1-score. Дополнительно проводится A/B-тестирование конфигураций.

NFR-13 (Безопасность видеопотоков и данных)

  • Все медиапотоки и данные должны передаваться по зашифрованным каналам (TLS) и храниться в зашифрованном виде.
  • Доля сессий без шифрования - 0%.
  • Система должна обнаруживать попытки подмены клиентского приложения.

Верификация:
Автоматический аудит TLS конфигурации ежемесячно. Выполняется выборка >= 500 соединений. Проводится ежегодный пентест.

NFR-14 (Потребление ресурсов на клиенте)

  • Приложение прокторинга не должно создавать значимой нагрузки на устройство пользователя при штатной работе.

Верификация:
Тестирование на >= 3 конфигурациях в течение >= 30 минут. Измеряются CPU, RAM и FPS. Контролируется CPU <= 30%, RAM <= 700 МБ, FPS >= 12.

NFR-15 (Восстановление после сбоев - RTO/RPO)

  • RTO для критических функций - не более 1 часа.
  • RPO для экзаменационных данных - не более 10 минут.
  • Конфигурационные данные должны восстанавливаться без существенной потери.

Верификация:
DR-тестирование >= 2 раза в год. Выполняется симуляция отказа кластера. Замеряются RTO и RPO на выборке >= 100 сессий.
