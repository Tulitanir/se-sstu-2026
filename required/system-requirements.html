Системные требования (SR-SYS)

SR-SYS-1 (Сценарий 1 - Переход в SSO)

При инициировании пользователем входа в систему, система должна перенаправлять пользователя в систему аутентификации (SSO).

Верификаиция: функциональный тест (инициирование входа => пользователь перенаправлен в SSO)

SR-SYS-2 (Сценарий 1 - Успешный вход в систему)

При успешной аутентификации пользователя в SSO система должна принимать и обрабатывать токен доступа, возвращённый SSO.

Верификация: функциональный тест (SSO возвращает валидный токен => система обрабатывает токен и продолжает процесс авторизации).

SR-SYS-3 (Сценарий 1 - Безуспешный вход в систему)

При безуспешной аутентификации пользователя в SSO система не должна создавать пользовательскую сессию, уведомив пользователя об ошибке аутентификации.

Верификация: функциональный тест (SSO возвращает ошибку аутентификации => пользователь не авторизован => отображено сообщение об ошибке)

SR-SYS-4 (Сценарий 1 - Назначение ролей)

При успешной аутентификации пользователя система должна определять и назначать ему роль на основании данных, полученных от SSO.

Верификация: функциональный тест (получен токен с атрибутами пользователя => система извлекает роль => пользователю доступны функции, соответствующие роли)

SR-SYS-5 (Сценарий 1 - Истечение токена: без повторной аутентификации)

При обнаружении истечения срока действия токена доступа пользователя система должна инициировать процесс получения нового токена через SSO и, при успешном получении нового токена, продолжать текущую пользовательскую сессию без повторной аутентификации.

Верификация: функциональный тест (истёкший токен => система выполняет запрос к SSO => получает новый токен => пользователь сохраняет авторизованную сессию без повторного входа)

SR-SYS-6 (Сценарий 1 - Истечение токена: с необходимостью повторной аутенфикации)

При невозможности автоматического получения нового токена через SSO система должна прекращать выполнение защищённых операций и инициировать повторную аутентификацию пользователя.

Верификация: функциональный тест (новый токен не получен => блокировка защищённых операций => редирект на SSO)

SR-SYS-7 (Сценарий 1 - Заблокированный пользователь)

Если пользователь имеет статус “заблокирован”, система должна отклонять авторизацию и запрещать выполнение любых операций от имени пользователя, даже при успешной аутентификации в SSO.

Верификация: функциональный тест (пользователь с валидными SSO-данными, но статусом “blocked” => сессия не создается, доступа к системе нет => сообщение о блокировке)

SR-SYS-8 (Сценарий 1 - Недоступность SSO)

При недоступности системы аутентификации (SSO) система должна запрещать выполнение операций, требующих полной аутентификации пользователя, и обеспечивать доступ только для пользователей с ранее установленной и действующей локальной сессией

Верификация: функциональный тест: SSO недоступен:

  • пользователь без локальной сессии => доступ запрещён
  • пользователь с валидной локальной сессией => ограниченный доступ предоставлен

SR-SYS-9 (Сценарий 2 - Допуск к старту экзамена)

Система должна разрешать запуск экзамена только авторизованному студенту в пределах открытого окна экзамена.

Верификация: функциональный тест (авторизованный студент + наступило время экзамена => запуск разрешён).

SR-SYS-10 (Сценарий 2 - Проверка попыток и доступа к экзамену)

При попытке запуска экзамена система должна проверять, что пользователь имеет право на прохождение экзамена и что количество доступных попыток не исчерпано, после чего либо разрешать запуск экзамена, либо отклонять его запуск.

Верификация: функциональный тест (нет доступа или попытки исчерпаны => запуск отклонён, есть доступ и попытки не исчерпаны => запуск разрешен).

SR-SYS-11 (Сценарий 2 - Запрос доступа к камере и микрофону)

Перед началом экзамена система должна инициировать запрос разрешений на доступ к камере и микрофону пользователя, необходимых для проведения прокторинга.

Верификация: функциональный тест (пользователь нажимает “Начать экзамен” => появляется запрос доступа к камере и микрофону).

SR-SYS-12 (Сценарий 2 - Проверка и блокировка старта без устройств)

Система должна разрешать начало экзамена только при наличии доступа к камере и микрофону, и должна блокировать запуск экзамена при отсутствии хотя бы одного из требуемых разрешений или недоступности устройств.

Верификация: функциональный тест (доступ к камере/микрофону отсутсвтует => старт заблокирован).

SR-SYS-13 (Сценарий 2 - Создание прокторинг-сессии)

После успешного прохождения всех проверок допуска к экзамену (включая аутентификацию, проверку попыток и доступность необходимых устройств) система должна создавать уникальную прокторинг-сессию, связанную с конкретным пользователем и экзаменом.

Верификация: функциональный тест (пользователь прошёл все проверки допуска => выполнено “Начать экзамен” => создана прокторинг-сессия).

SR-SYS-14 (Сценарий 2 - Подключение видеопотока)

После создания прокторинг-сессии и получения необходимых разрешений пользователя система должна устанавливать соединение с сервисом передачи медиапотоков и передавать видеопоток, связанный с экзаменационной сессией пользователя.

Верификация: интеграционный тест (создана прокторинг-сессия => видеопоток успешно передаётся в media-сервис).

SR-SYS-15 (Сценарий 2 - Включение античита)

После создания прокторинг-сессии система должна инициализировать античит, передавая идентификатор экзаменационной сессии и параметры наблюдения, для последующего анализа пользовательской активности в ходе экзамена.

Верификация: интеграционный тест (создана прокторинг-сессия => античит получает session_id и параметры конфигурации => античит инициализирован).

SR-SYS-16 (Сценарий 2 - Выдача вопросов и запуск таймера)

После успешного завершения всех этапов инициализации экзаменационной сессии система должна загрузить и отобразить студенту набор экзаменационных заданий и одновременно инициировать запуск серверного таймера экзамена.

Верификация: функциональный тест (экзамен начат => отображаются вопросы, запущен таймер).

SR-SYS-17 (Сценарий 2 - Фиксация записи камеры, микрофона и экрана)

Во время активной экзаменационной сессии система должна обеспечивать захват и сохранение медиапотоков (видеопоток с камеры, аудиопоток с микрофона и поток экрана пользователя) с привязкой к идентификатору экзаменационной сессии.

Верификация: функциональный тест (экзамен заупщен => создаются записи всех предусмотренных источников).

SR-SYS-18 (Сценарий 2 - Сохранение результатов вместе с записями)

После завершения экзаменационной сессии система должна сохранять результаты экзамена, медиозаписи (камера, микрофон, экран) и журнал событий прокторинга и античита как единый связанный набор данных, ассоциированный с идентификатором экзаменационной попытки.

Верификация: функциональный тест (экзамен завершен => сохранены ответы, видео и аудио, события прокторинга и связаны с уникальным индентификатором экзаменнационной сессии).

SR-SYS-19 (Сценарий 2 - Потеря интернет-соединения)

При потере сетевого соединения во время активной экзаменационной сессии система должна переводить экзамен в оффлайн-режим, обеспечивать локальное сохранение вводимых ответов и состояния таймера, а также сохранять возможность продолжения выполнения экзамена без прерывания пользовательской сессии.

Верификация: функциональный тест (разрыв соединения => система не завершает экзамен, сохраняет ответы локально,отображает статус offline).

SR-SYS-20 (Сценарий 2 - Недоступность стриминг-сервиса)

При недоступности SFU-сервиса система должна автоматически переключаться в режим локальной записи медиапотоков (камера, микрофон, экран) с их временным хранением на устройстве пользователя и последующей автоматической загрузкой в систему после восстановления соединения с SFU.

Верификация: интеграционный тест (SFU недоступен => запись продолжается локально и буферизует медиафайлы до момента восстановления).

SR-SYS-21 (Сценарий 2 - Недоступность SSO)

Если недоступность SSO возникает во время активной экзаменационной сессии, система должна обеспечивать непрерывность выполнения экзамена, включая ввод и сохранение ответов, без прерывания сессии пользователя, при условии действующей локальной экзаменационной сессии.

Верификация: функциональный тест:

  • пользователь начал экзамен
  • SSO становится недоступен
  • Система не завершает экзамен, продолжает отображать задания, сохраняет ответы, пользователь может завершить экзамен

SR-SYS-22 (Сценарий 3 - Регистрация подозрительного события)

При обнаружении подозрительного события любым компонентом системы (античит, прокторинг или анализ медиопотоков) система должна создавать запись инцидента нарушения, содержащую тип события, время фиксации, идентификатор экзаменационной сессии и ссылку на связанные артефакты (видео, экран, логи).

Верификация: функциональный тест (произвести моделирование подозрительного события => создается запись о событии с указанием типа, времени фиксации, идентификатора экзаменационной сессии и ссылкой на связанные артефакты).

SR-SYS-23 (Сценарий 3 - Уведомление проктора)

После регистрации инцидента система должна отправлять проктору уведомление в режиме близком к реальному времени, содержащее идентификатор экзаменационной сессии, тип события и ссылку на связанные артефакты (видео/экран/логи).

Верификация: функциональный тест (создаётся запись об инциденте => проктор получает уведомление).

SR-SYS-24 (Сценарий 3 - Доступ проктора к фрагменту записи)

Для зарегистрированного инцидента нарушения система должна предоставлять проктору доступ к связанному медиофрагменту, включающему видеопоток с камеры и синхронизированные аудио- и экранные данные, охватывающему временной интервал до и после момента фиксации события.

Верификация: функциональный тест (получено уведомление об инциленте => проктор открывает инцидент => система отображает видеофрагмент и временную шкалу события)

SR-SYS-25 (Сценарий 3 - Фиксация решения проктора)

Система должна позволять проктору фиксировать решение по зарегистрированному инциденту с указанием типа решения, комментария и подтверждающих артефактов. Решение должно быть привязано к инциденту и сохраняться в системе в неизменяемом виде с записью автора и времени принятия.

Верификация: функциональный тест (существует запись инцидента => проктор принимает решение => система сохраняет тип решения, время решения, идентификатор проктора, ссылку на инцидент).

SR-SYS-26 (Сценарий 3 - Подтверждённое нарушение)

Если проктор принимает решение о подтверждении нарушения, система должна отправить студенту уведомление, содержащее тип нарушения, время фиксации, комментарий проктора.

Верификация: функциональный тест (проктор подтверждает нарушение => студент получает уведомление).

SR-SYS-27 (Сценарий 3 - Массовые события)

При поступлении большого количества событий от систем мониторинга система должна агрегировать события, оценивать их уровень риска и формировать упорядоченную очередь инцидентов для обработки проктором на основе приоритета.

Верификация: нагрузочный/функциональный тест (создается N событий => система агрегирует связанные события в инциденты, присваивает уровень риска, формирует очередь приоритетов).

SR-SYS-28 (Сценарий 3 - Режимы деградации при недоступности проктора)

Если проктор недоступен в момент или после завершения экзаменационной сессии, система должна переводить результаты экзамена в состояние “ожидает проверки” и обеспечивать возможность принятия решения по результатам экзамена на основе сохранённых артефактов (ответы студента, медиозаписи и журналы событий).

Верификация: функциональный тест (проктор отсутствует и экзамен завершен => система сохраняет все артефакты и устанавливает статус “требует проверки”).

SR-SYS-29 (Сценарий 4 - Доступ администратора к настройкам прокторинга)

Система должна предоставлять доступ к разделу “Настройки прокторинга” только пользователям с ролью администратора учебного продразделения, проверяя права доступа на уровне пользовательской сессии и серверной авторизации. Все попытки доступа должны проходить проверку прав перед выполнением операции.

Верификация: функциональный тест (пользователь с ролью “Администратор уч. подразделения” имеет доступ к настройкам. Пользователь с иной ролью такого доступа не имеет).

SR-SYS-30 (Сценарий 4 - Настройка допустимых переключений вкладок)

Система должна позволять администратору учебного подразделения задавать максимальное допустимое количество переключений вкладок в рамках одной экзаменационной сессии, которое используется системой прокторинга для фиксации потенциальных нарушений и формирования событий риска.

Верификация: функциональный тест (администратор устанавливает значение => система сохраняет параметр => параметр применяется к новым экзаменам).

SR-SYS-31 (Сценарий 4 - Настройка срока хранения записей)

Система должна позволять администратору учебного подразделения задавать срок хранения экзаменационных данных (включая видеозаписи, аудиозаписи, экранные записи, ответы студентов и журналы событий).

Верификация: функциональный тест (администратор задаёт срок хранения => система сохраняет настройку).

SR-SYS-32 (Сценарий 4 - Политика хранения данных)

Система должна автоматически обеспечивать удаление или обезличивание экзаменационных данных по истечении установленного срока хранения.

Верификация: функциональный тест (истек срок хранения => данные становятся недоступны, выполняется удаление или обезличивание).

SR-SYS-33 (Сценарий 4 - Настройка параметров античита)

Система должна позволять авторизованному администратору учебного подразделения изменять ограниченный набор параметров античит-системы, используемых для обнаружения подозрительных действий в рамках экзаменационных сессий. Изменения должны фиксироваться, проходить валидацию и гарантированно доставляться в античит-сервис с подтверждением применения.

Верификация: функциональный тест (администратор изменяет параметр => система валидирует значение, сохраняет конфигурацию и отправляет в античит-сервис => античит получает параметр и использует в новых сессиях).

SR-SYS-34 (Сценарий 4 - Сохранение и применение новых политик)

Система должна обеспечивать версионирование политик прокторинга и применять актуальную версию политики только к экзаменационным сессиям, созданным после момента её публикации, сохраняя неизменность ранее назначенных политик для уже созданных экзаменов.

Верификация: функциональный тест:

  1. создаётся политика v1 => создаётся экзамен A => экзамен A использует v1
  2. изменяется политика => создаётся v2 => создаётся экзамен B => экзамен B использует v2
  3. Проверка:
  • экзамен A всегда использует v1
  • экзамен B всегда использует v2
  • изменение v2 не влияет на экзамен A

SR-SYS-35 (Сценарий 5 - Доступность подачи апелляции)

Система должна предоставлять студенту возможность подать апелляцию только по экзаменационной сессии, находящейся в статусе “завершена и оценена”, и только в пределах установленного временного окна после публикации результата.

Верификация: функциональный тест:

  1. экзамен в статусе “в процессе” или “ожидает проверки” => пелляция недоступна
  2. экзамен оценён => апелляция доступна
  3. истекло время подачи апелляции => апелляция недоступна

SR-SYS-36 (Сценарий 5 - Обязательность указания причины)

При подаче апелляции система должна создавать уникальную запись апелляции, связанную с результатом экзаменационной сессии, фиксировать обязательную причину обращения, и устанавливать начальный статус “зарегистрирована”.

Верификация: функциональный тест:

  1. попытка отправить апелляцию без причины => отклоняется системой
  2. попытка отправить с причиной => апелляция создаётся, причина фиксируется в неизменном виде, записи апелляции назначается уникальный индентификатор и статус “зарегистрирована”
